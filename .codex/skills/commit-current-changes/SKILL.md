# Commit Current Changes

以下の手順で、既に存在する作業ツリーの変更を確認し、作業単位として整理してコミットする。

## 目的

作業済みの変更内容を壊さずに把握し、適切なブランチ名、`docs/tasks/` の記録、コミットを作成する。

## 前提

- 既存変更はユーザーまたは別作業の成果物として扱い、勝手に戻さない。
- `git push` は行わない。push が必要な場合はユーザーの明示依頼を待つ。
- 作業ツリーに依頼範囲外の変更が混在する場合は、対象ファイルを切り分けてステージする。
- 変更内容が不明確、破壊的、または複数の独立作業に分かれる場合は、コミット前にユーザーへ確認する。

## 手順

1. 現在のブランチと差分を確認する。

```bash
git branch --show-current
git status --short --branch
git diff --stat
git diff
```

2. 差分から作業タイトルとブランチ名を決める。
   - ブランチ名は `fix/` `feature/` `refactor/` `docs/` `env/` のいずれかの接頭辞を使う。
   - アンダースコア `_` は使わず、単語区切りはハイフン `-` にする。
   - 既に適切な作業ブランチ上にいる場合は、新規ブランチを作らず現在ブランチを使う。
   - `main` 上にいる場合は、未コミット変更を保持したまま `git checkout -b <branch-name>` で作業ブランチを作成する。

3. `docs/tasks/` に作業記録を作成する。

```bash
date +"%Y%m%d_%H%M"
```

ファイル名は `yyyyMMdd_hhmm_タイトル.md` とする。タイトルに作業分類の接頭辞は含めない。

記載内容:

- 作業タイトル
- 確認した現在変更の概要
- 対象ファイル
- 検証内容
- コミット予定内容

4. 必要な検証を行う。
   - 変更対象の種類に応じて、プロジェクトの lint、型チェック、テスト、ビルドを確認して実行する。
   - 実行できない検証がある場合は理由をタスクファイルまたは最終報告に記録する。

5. 差分を再確認し、対象ファイルだけをステージしてコミットする。

```bash
git diff
git status --short --branch
git add <target-files>
git commit -m "<変更内容がわかる具体的な日本語メッセージ>"
git status --short --branch
```

## 失敗時

ブランチ作成、`git add`、コミットなど `.git/` 更新を伴う操作が権限またはロックで失敗した場合は、同じ通常権限コマンドを繰り返さず、権限昇格またはユーザー側での実行に切り替える。

作業中にバグや環境問題が発生した場合は、`AGENTS.md` のルールに従って `docs/issues/` にレポートを作成する。
