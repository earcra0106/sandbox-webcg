# Codex Notes

このリポジトリは、WebGL と WebGPU を使ってウェブページ上で CG を描画するための実験プロジェクトです。

## プロジェクト方針

- WebGL と WebGPU の違いを比較しやすい実験コードを蓄積する。
- 1つの描画テーマに対して、可能な範囲で WebGL 版と WebGPU 版を並べて実装する。
- 描画ロジック、シェーダー、初期化処理、リソース管理の違いが追いやすい構成にする。
- 実験コードであっても、ブラウザでの動作条件、制限、失敗時の挙動を明記する。

## 実装時の注意

- WebGPU は対応ブラウザや実行環境の制約があるため、`navigator.gpu` の存在確認とフォールバック表示を用意する。
- WebGL は WebGL 2 を優先し、必要があれば WebGL 1 との差分を明記する。
- シェーダーコードは、GLSL と WGSL の対応関係が分かるように配置またはコメントする。
- サンプルを追加した場合は、目的、操作方法、対応 API、既知の制限をドキュメントに追記する。
- UI は実験結果を確認するためのツールとして扱い、過度な装飾よりも描画領域とパラメータ操作を優先する。

## 作業ルール

作業中には以下のルールを守ること。

1. 今後実行環境に関する情報提供があったり、環境構築中にインストールした主要パッケージについては、必ず `docs/specs/` に記載する。トークンの肥大化を避けるため、各パッケージごとに記載するファイルを分割し、 `docs/PROJECT_SPEC.md` をそれらの目次にすること。

2. 作業を始める前に、要件定義を行う。以下の内容を含めること。なおドキュメント更新のみの作業では作業方針の記述のみで構わない。
    - 作業タイトル
        - 作業タイトルはgit branch名としても利用するため、以下のルールに従い命名する。
            - ブランチ名の最初に、作業分類ごとに `fix/` `feature/` `refactor/` `env/` の接頭辞を付ける。
            - ブランチ名にアンダースコア (_) は使用しない。単語の区切りはハイフン (-) を利用する。
    - 作業の目的と概要
    - 作成・編集するファイルのスコープ
    - 実装する関数、クラス、モジュールの概要
    - 実装した機能にアクセスするためのURLパスやUI操作の説明

3. 作業開始時に以下を行うこと。
    - 要件定義の内容を `docs/tasks/` に記載すること。ファイル名は `yyyyMMdd_hhmm_タイトル.md` とすること。ここではタイトルに作業分類の接頭辞を含めないこと。またファイル名用の作業時刻は `date +"%Y%m%d_%H%M"` コマンドで取得すること。
    - 現在のブランチが `main` であることを確認する。
    - 作業タイトルをもとに、 `git checkout -b ブランチ名` コマンドで作業ブランチを作成する。
    - ブランチ作成に失敗した場合は、`docs/issues/` に原因と代替案を記録し、ユーザー確認の上で作業を継続する。

4. 作業を適切に分割してコミットすること。コミットメッセージは、変更内容がわかるように具体的に日本語で記述すること。`git push` などのリモート操作は、ユーザーが明示的に許可または依頼した場合のみ行うこと。

5. 作業中にバグが発生した場合は、タイトル、発生日時( `yyyy/MM/dd hh:mm` )、分類、原因、解決方法、解決できない場合は代替案などを記載したレポートを作成すること。発生日時は `date +"%Y/%m/%d %H:%M"` コマンドで取得すること。
    - 作成したレポートを `docs/issues/` フォルダの下に、 `yyyyMMdd_hhmm_タイトル.md` として配置すること。
    - バグの分類は以下から選択し、適切な表現がなければ、この文書に追加すること。
        - `env` : 環境構築や実行環境に関する問題
        - `build` : ビルドプロセスや依存関係に関する問題
        - `code` : コードのバグやロジックエラー
        - `performance` : パフォーマンスの問題
        - `compatibility` : ブラウザやプラットフォーム間の互換性の問題
        - `other` : 上記以外の問題
